## What Went Well

**Amelia (Developer):** "We delivered the entire happy-path app — scaffold to ship-ready binary — in 8 cohesive stories, with zero abandoned work and zero regressions across stories."

### 1. Story Decomposition Was Right-Sized

The split into 8 stories matched the natural seams of the architecture: scaffold → persistence → API → frontend foundation → input component → list/CRUD → production polish → tests. Each story had a clear, testable outcome. **No story had to be split or re-scoped mid-flight.**

### 2. Cross-Story Handoffs Were Clean

**Charlie (Senior Dev):** "The `data-slot` placeholder pattern in Story 1.4 was the standout decision. Stories 1.5 and 1.6 dropped into named slots; the handoff was unambiguous."

Examples:
- Story 1.4's `<div data-slot="task-input" />` and `<div data-slot="task-list" />` placeholders ↔ Story 1.5/1.6's drop-in components.
- Story 1.5's `onSubmit(text: string)` callback contract ↔ Story 1.6's `useTasks.createTask` consumer (no re-trim, no UUID re-gen, no length re-validation needed).
- Story 1.6's exposed-but-unused `loadError` field ↔ Story 2.2's planned page banner (already wired through, just needs the UI).

### 3. NFR Headroom Maintained Throughout

**Alice (Product Owner):** "We didn't blow any of the NFR budgets. Each story had measurable headroom against the next."

- LOC: ended at 886 / 1000 (114 free)
- Bundle: 74.5 KB / 100 KB (27 KB free)
- Prod deps: 10 / 10 (at cap, but every dep justified per architecture)
- API p95: 0.5 ms / 300 ms (600× margin)

### 4. Spec-Mandated Hard Rules Held

**Dana (QA Engineer):** "Every story preserved the rules from the architecture doc — even when the spec sketches got it slightly wrong, the underlying invariants held."

- `better-sqlite3` imported only in `db.ts` (verified by grep in every code review)
- snake_case ↔ camelCase conversion only in `db.ts`
- One-way import direction: `server.ts → routes/ → db.ts` and `components → hooks → state → api`
- No `dangerouslySetInnerHTML` anywhere (NFR-S1)
- Reducer purity (no fetch/Date.now/randomUUID inside) — paid huge dividends in Story 1.8's tests

### 5. Pre-emptive Carve-Outs Saved Re-Litigation

**Charlie (Senior Dev):** "Both NFR-M3 carve-outs (shadcn primitives in 1.7, test files in 1.8) were documented in the story's Dev Notes BEFORE the dev started, so the code review didn't have to argue them post-hoc."

The pattern: when a story knew it would need a defensible interpretation of an NFR carve-out, the story spec called it out up front with the reasoning. Reviewers accepted the reasoning rather than re-debating the count.

## What Was Challenging

### 1. Spec Drift From Tooling Reality

**Charlie (Senior Dev):** "Multiple stories had to deviate from the spec wording because the spec was written against an idealized toolchain."

Concrete examples (each documented as an "accepted deviation" in Completion Notes):

| Story | Spec said | Reality | Resolution |
|---|---|---|---|
| 1.3 | `node --env-file=.env` "warns and continues" when missing | Plain `--env-file` hard-errors | Used `--env-file-if-exists` (silent variant) |
| 1.3 | Wrap security headers in `app.register(securityHeaders)` | Fastify plugin encapsulation scopes hooks to children only — sibling plugins (`tasksRoutes`) never received headers | Direct `registerSecurityHeaders(app)` call (no plugin wrapper, no `fastify-plugin` dep) |
| 1.3 | Fastify defaults handle AC4 (extra props rejected, types rejected) | Fastify silently strips extra props AND coerces types | Added `ajv: { customOptions: { removeAdditional: false, coerceTypes: false } }` |
| 1.4 | Individual `@radix-ui/react-checkbox`, `@radix-ui/react-label`, `@radix-ui/react-slot` packages | Modern shadcn install uses consolidated `radix-ui` package | Followed shadcn CLI's actual output |
| 1.4 | Nova preset gives clean defaults | Nova preset adds Geist web font (violates AC5) | Removed `@fontsource-variable/geist` post-install |
| 1.6 | Checkbox tick scales 0→1 in 100 ms `ease-out` (AC5) | shadcn `<CheckboxPrimitive.Indicator>` ships `transition-none` and only mounts when checked | Deferred to Story 2.6; `forceMount` + custom CSS would require editing the primitive (forbidden) |

**Lesson:** Future story specs should reserve a margin for "tooling reality may differ from documented behavior". Specs that prescribe exact tool invocations (e.g., `npx shadcn@latest init` output) drift fastest.

### 2. Code-Reviewer False-Alarm Rate Was High

**Amelia (Developer):** "The Blind Hunter layer — without project context — flagged a lot of issues that were either by-design or simply wrong."

False alarms across reviews included:
- `radix-ui` is a hallucinated package (it isn't; modern shadcn uses it)
- `max-w-150` is non-standard Tailwind (it isn't; v4 spacing scale)
- `lucide-react ^1.11.0` is a wrong major (it isn't in 2026)
- Node 24 / TypeScript 6 / Vite 8 versions don't exist (they do in 2026)
- `--env-file` overrides inline `NODE_ENV` (Node's documented behavior is the opposite)
- `crypto.randomUUID()` throws on non-secure contexts (true, but the deployment model is HTTPS / localhost)
- UTF-16 vs SQLite `length()` length mismatch creates a 5xx leak (empirically falsified — UTF-16 units are always ≥ codepoints)

**Acceptance Auditor (with spec access) was consistently the most reliable layer.** Edge Case Hunter (with file-system access) caught most of the real issues. Blind Hunter was useful for the few times its skepticism caught a genuine bug, but its noise floor was high.

**Lesson:** When triaging code-review findings, prioritize Acceptance Auditor and Edge Case Hunter findings; treat Blind Hunter as a supplementary noise channel where the signal-to-noise ratio depends on the story's complexity. The Blind Hunter's most valuable contribution was cross-validation when it independently flagged the same issue another reviewer had spotted (e.g., the row-vs-span opacity issue in Story 1.6).

### 3. Module-Singleton Patterns Are Theoretically Fragile

**Edge Case Hunter** flagged this in multiple reviews: `db.ts` opens its SQLite connection at module import time. This works because:
- Tests use `process.env.DB_PATH = ...` BEFORE `await import("./db.js")` to override the path.
- `tsx --test` runs each test file in a separate child process by default.

But the pattern silently breaks if anyone ever:
- Switches test runner to in-process isolation
- Imports both `db.test.ts` and `routes/tasks.test.ts` from a single entry
- Adds a third test file that imports `db.ts` differently

**Lesson:** The factory pattern (`openDb(path: string)`) would eliminate this fragility. Deferred to Epic 2 if/when it bites; for now, the test files explicitly use distinct `DB_PATH` filenames per file as defensive coding.

### 4. Story 1.4 Was Disproportionately Risky

**Alice (Product Owner):** "Story 1.4 was the only story that fundamentally changed the project's shape — replaced the scaffold, added 7 prod deps, switched the styling architecture from CSS Modules to Tailwind."

All subsequent stories (1.5, 1.6, 1.7, 1.8) were composition or thin slices. The risk concentration in 1.4 was justified (it's the natural shape of "set up the design system once, then build on it") but worth noting: had 1.4's deviations been blocking, the whole frontend half of Epic 1 would have stalled.

**Lesson:** When a single story carries this much architectural risk, the story spec should be explicit about which decisions are reversible vs. load-bearing for downstream stories. Story 1.4's spec did this well — the three flagged deviations were all anticipated.

### 5. Several Polish Items Accumulated as Defers

**Dana (QA Engineer):** "Each review surfaced a few items that were 'real but not in scope'. They piled up."

Notable defers that should be addressed in Epic 2:
- Focus restoration after row delete (Story 1.6 review)
- Checkbox tick scale animation (Story 1.6 — AC5 not literally met without primitive edit)
- Server eslint config (Story 1.8 — pre-existing Story 1.1 shape)
- Security-headers regression test (Story 1.7 + 1.8 reviews — both deferred)
- AR25 route-ordering smoke test (same)
- aria-live verbosity on `<ul>` (Story 1.6 review — spec mandated, may need tuning)
- `ROLLBACK` action concurrent-mutation stomping (Story 1.6 — explicit Story 2.3 replacement target)

These cluster around **a11y / quality** and **resilience under failure**, both of which are Epic 2's themes. Story 2.6 (a11y/quality QA pass) is the natural landing spot.

## Key Insights

1. **Specs that document the hard *invariants* (snake_case stays in db.ts, no dangerouslySetInnerHTML, reducer purity) outperform specs that prescribe exact tool incantations.** The invariant-style rules held across all 8 stories with zero breakage. The tool-incantation specs (e.g., "use individual `@radix-ui/react-*` packages") all needed deviation.

2. **Pre-emptive carve-out documentation in Dev Notes saves review time.** Both NFR-M3 carve-outs (shadcn primitives, test files) were explained up front in the story spec, so reviewers accepted the count rather than re-litigating.

3. **The optimistic-UI architecture from Story 1.6 paid off in Story 1.8.** Reducer purity (no fetch / no Date.now / no randomUUID inside) made the reducer trivially testable: `Object.freeze` the input, dispatch, assert no mutation. Discipline at design time → cheap tests at verification time.

4. **The `data-slot` placeholder pattern is reusable.** Story 1.4 introduced it; Stories 1.5 and 1.6 consumed it cleanly. Worth keeping as a convention for future "scaffold first, fill in later" patterns.

5. **Code-review false-alarm filtering benefits from spec access.** Reviewers without context (Blind Hunter) generated 5–10× more noise than reviewers with the spec (Acceptance Auditor). This is a cost-benefit calibration for which review layers to prioritize.
