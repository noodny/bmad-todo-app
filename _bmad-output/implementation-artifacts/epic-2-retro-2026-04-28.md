## What Went Well

**Amelia (Developer):** "Epic 2 was a much harder shape than Epic 1 — every story added behavior that had to coexist with the Epic-1 happy path without breaking it. We landed all 6 stories with zero regressions and zero new prod deps."

### 1. Forward-Engineered Composition Paid Off (Three Times)

The compositional `<PageBanner>` design from Story 2.2 — `icon` / `message` / `action` slots instead of a `variant` enum — paid dividends in Stories 2.4 and 2.5:

- **Story 2.4** dropped in the offline variant in 5 lines, no PageBanner edits.
- **Story 2.5** reused PageBanner verbatim for the ErrorBoundary fallback (icon + message + Reload action).

The Story 2.4 spec literally said "discriminated `variant` prop"; the implementation chose to interpret that liberally and stay compositional. Story 2.6 confirmed AC11 was satisfied: a single `PageBanner.tsx` file handled all three variants, no `OfflineBanner`/`LoadFailedBanner`/`ErrorBoundaryBanner` forks. **The forward-design judgment from Story 2.2 saved ~30 LOC of duplicate code over the rest of the epic.**

### 2. The `runMutation` Helper Pattern Was the Right Refactor at the Right Time

Story 2.4's Task 1 reclamation introduced `runMutation` to fund the offline-banner LOC budget. By consolidating the three mutation callbacks (createTask, toggleTask, deleteTask) plus retryMutation through a single helper, it:

1. Reclaimed ~10 LOC for Story 2.4's additions.
2. Centralized the `instanceof TypeError` network-vs-HTTP discrimination in ONE place (the catch handler) rather than 4.
3. Made Story 2.5's NFR-R4 audit trivial — every catch path goes through the same logger call, so the silent-swallow grep was guaranteed clean.

Charlie's call to extract this rather than inline the network-detect logic in each mutation was a load-bearing decision for the rest of the epic.

### 3. Soft-Delete Was the Cleanest Replacement for ROLLBACK

Story 2.3's hardest design decision was how to handle "row was visually removed by optimistic delete, but the DELETE failed → row should reappear." The two options:

- **Snapshot the deleted task in the hook** + new `RESTORE_DELETED` action (essentially ROLLBACK by a different name).
- **Soft-delete via `pendingMutation: 'delete'` + render-layer filter** — the row stays in `state.tasks` marked-pending; TaskList hides it; SYNC_OK removes; SYNC_FAIL flips to `'failed'` so the row reappears.

The team chose soft-delete. It cost +1 LOC in the reducer (filter → map) and +1 LOC in TaskList (the filter), but eliminated the snapshot-ref pattern that ROLLBACK required. It also made `retryMutation` trivial — the task data was already in state.

**Amelia (Developer):** "The reducer stayed the single source of truth. AR21 purity was preserved. Code-review's high-severity findings on Story 2.3 (concurrent retry races) all resolved cleanly with one ref + the `retryInFlightRef` guard."

### 4. NFR Discipline Held Under Real Pressure

Every Epic-2 story crashed the LOC budget at least once during dev:

| Story | LOC trajectory |
|---|---|
| 2.1 | 886 → 897 (no reclamation needed) |
| 2.2 | 897 → 1001 (1-LOC bust) → 993 (compressed App.tsx destructure) |
| 2.3 | 998 → 1040 (busted by 40!) → 989 (`mapTask` reducer helper saved 51 LOC) |
| 2.4 | 998 → 962 (Task 1 reclamation, −33) → 995 (additions absorbed exactly) |
| 2.5 | 998 → 1006 (busted by 6) → 998 (ErrorBoundary inline-method compression + App.tsx Button JSX inline) |
| 2.6 | 998 → 1009 (Phase B1 +11) → 994 → 997 (after review patch) |

NFR-M3 held in every story's final state. The discipline pattern was: **add code → measure LOC → reclaim from existing comments / JSX collapse → repeat until under cap**. Story 2.3's `mapTask` reducer helper was the standout — turned 5 multi-line cases into single-liners and reclaimed 51 LOC in one move.

**Charlie (Senior Dev):** "The cap forced us to be honest about every line. By Story 2.5, comment-density was way down — but the *why* comments survived. The trims were the right ones."

### 5. The Race-Condition Catches Were the High-Value Moments

Code review consistently surfaced timing issues that wouldn't have shown up in tests:

- **Story 2.2:** double-click Retry race on `retryInitialLoad` → fixed via `loadCleanupRef`.
- **Story 2.3:** concurrent retry race on `retryMutation` → fixed via `retryInFlightRef`.
- **Story 2.4:** stuck-offline after transient TypeError → fixed by dispatching `CONNECTIVITY_CHANGED({online:true})` on every successful `SYNC_OK` + idempotent reducer case.
- **Story 2.5:** logger flush race + handler-registration-too-late → combined fix via `fatalExit` helper with `setImmediate` + relocation to right after Fastify factory.

These were ALL flagged by `bmad-review-edge-case-hunter` running in parallel with Blind Hunter and Acceptance Auditor. Without the parallel review, four shippable-but-fragile bugs would have made it through.

### 6. Story 2.6's Honest Partial-Completion Was the Right Move

Story 2.6's verification pass requires browsers, screen readers, and real iOS/Android devices — none available in a CLI dev environment. The dev-agent could have:

1. Claimed AC1–AC11 passed without evidence, OR
2. Generated a placeholder VERIFICATION.md with explicit `🟡 TBD-by-reviewer` per-AC repro steps.

It chose option 2 and was transparent about what was vs. wasn't performed. The 10 TBD rows have explicit instructions; the 3 ✅ rows have programmatic evidence (curl+sqlite for AC13, static CSS read for AC6, doc creation for AC12). **Phase B Task B1 (focus-after-delete) was the one piece of automatable a11y work that the dev-agent could and did deliver** — closing the longest-deferred item in the backlog (originally surfaced in Story 1.6's review).

## What Was Challenging

### 1. The LOC Cap Became a Forcing Function — Sometimes Helpful, Sometimes Distorting

**Charlie (Senior Dev):** "Every story crashed the cap. Most were saved by trimming dev-comments — that's mostly fine, but Story 2.3 had to introduce `mapTask` (a real abstraction) under budget pressure, not because it was the right architectural moment."

The compressions became their own pattern:

- Story 2.2: collapse App.tsx destructure to one line.
- Story 2.3: introduce `mapTask` reducer helper.
- Story 2.4: introduce `runMutation` hook helper.
- Story 2.5: inline ErrorBoundary class methods to single lines; collapse App.tsx Button JSX.
- Story 2.6: collapse App.tsx `<TaskList>` JSX prop list.

Some of these are clean wins (`mapTask`, `runMutation`). Others are pure compression (single-line class methods, JSX prop lists) that may be slightly harder to scan in editors. **Lesson for Epic 3 (if it happens):** if the budget is genuinely unworkable, propose raising NFR-M3 or carving out a per-component sub-cap. Don't optimize for the metric to the detriment of readability.

### 2. Browser-Required ACs in Story 2.6 Couldn't Be Performed by the CLI Dev-Agent

Ten of Story 2.6's thirteen ACs require an interactive browser. In a CLI session, the dev-agent had to either fake them or document the gap honestly. The "honest gap" approach (`🟡 TBD-by-reviewer` rows in VERIFICATION.md) is correct, but it shifts work to the human reviewer to actually run axe-core, Lighthouse, screen readers, viewport sweeps, and real-device testing.

**Alice (Product Owner):** "Until those rows are filled in by the human reviewer, we don't actually know if the app passes the a11y bar. The dev-story is `done`, but the verification isn't complete."

**Recommendation:** future projects with a similar verification-pass story should plan to involve a human reviewer with browser access in the loop. The dev-agent can do the static portion, the audit-tool installation, and the documentation skeleton; the reviewer must do the actual measurements.

### 3. The Deferred-Work Backlog Grew Faster Than It Shrank

| Epic | Items deferred in this epic | Items resolved in this epic | Net change |
|---|---|---|---|
| Epic 1 | ~36 | 0 (no resolution mechanism) | +36 |
| Epic 2 | ~21 | ~3 (focus-after-delete, B1 patch in 2.6; concurrency race in 2.4 review patch; uncaughtException defer-resolution from 1.2 in 2.5) | +18 |
| Total open at close | **79** | — | — |

Most deferred items are low-reachability (older JAWS variance, captive-portal `navigator.onLine` unreliability, theoretical TypeError catches) or pure polish (PageBanner fade-in animation, ErrorBoundary "Try again" reset path, comment-density restoration). None are shipping blockers per the static + automated verification.

But: 79 open items is a regret log that future maintainers will have to triage. **Recommendation for Epic 3+:** add a "deferred-work triage" task to each epic's retro to actively close items that are no longer relevant (e.g., a fix that was applied incidentally, or a constraint that no longer holds).

### 4. Code-Reviewer False-Alarm Rate Stayed High (Same as Epic 1)

Across the 6 Epic-2 reviews, ~46 findings were dismissed. Patterns of false alarms (consistent with Epic 1's retro):

- **Blind Hunter** (no project context) flagging spec-mandated patterns as anti-patterns: e.g., `aria-live="assertive"` + `role="alert"` redundancy (explicitly mandated by AC2 + UX-DR41).
- **Speculative concurrent-rendering concerns** (rAF vs React 19 commits) that don't manifest under React's batched dispatch.
- **Out-of-context fragility flags**: `getElementById("task-input")` magic string (established convention across the codebase since Story 1.6).

**Edge Case Hunter** continued to be the most reliable layer — its findings overlapped with real high-severity bugs at a much higher rate. **Acceptance Auditor** never produced a violation in Epic 2; only one partial deviation (Story 2.6 missing `verification/` directory, fixed in the same review).

**Pattern:** when triaging review findings, prioritize Edge Case Hunter and Acceptance Auditor; treat Blind Hunter as supplementary noise where the signal-to-noise improves only when independent reviewers concur.

### 5. The "Carve-Out to Story 2.6" Reflex Became a Crutch

By Story 2.5, "defer to Story 2.6" was the standard answer for any non-blocking concern. Story 2.6's deferred-work backlog inflated to ~30 items pre-execution, of which only 1 (Phase B Task B1) was actually picked up.

The remaining items defer further (now to "future maintenance"). This is fine per the spec — Story 2.6 is explicitly verification, not "fix everything in deferred-work" — but it does mean the regret log keeps growing. **Recommendation:** be more aggressive about closing items inline within the originating story when LOC headroom permits, rather than always deferring.

### 6. Story 2.6's CLI-Environment Limitation Highlighted a Workflow Gap

The dev-story workflow (`bmad-dev-story`) is designed for an agent that can read code, write code, and run shell commands. It cannot:

- Open a browser
- Run axe-core / Lighthouse against a live page
- Operate a screen reader
- Operate a real iOS or Android device
- Take screenshots

Story 2.6's spec acknowledges this implicitly (the manual-verification steps are documented for the human reviewer), but the dev-agent had to make that gap explicit in VERIFICATION.md. **Future workflow consideration:** a "browser-required verification" story type that explicitly halts and hands off to a human reviewer rather than letting the dev-agent produce a TBD document.

## Key Insights from Epic 2

### 1. Compositional UI > Variant Enums (When Multiple Stories Share the Surface)

The `<PageBanner>` design is the strongest example. Story 2.2 was forward-engineered for Story 2.4's offline variant + Story 2.5's ErrorBoundary fallback. The `variant` enum approach would have required edits in all three stories; the compositional slots required edits in zero (just new prop values).

**Generalization:** when designing a component that 2+ future stories will reuse, prefer slot-based composition over discriminated variants. The slots become the contract; downstream stories just pass different ReactNodes.

### 2. The Reducer Should Be the Single Source of Truth, Period

Every Epic-2 design decision that violated this rule (or considered violating it) ended up unwinding back to "put it in the reducer":

- Story 2.3 considered a hook-side `Map<id, () => Promise>` for retry closures → chose `pendingMutation` field on the task instead.
- Story 2.4 considered a hook-side `useState(boolean)` for online → chose `state.online` in the reducer with the hook as a thin event bridge.
- Story 2.5's ErrorBoundary state COULD have been duplicated in the reducer ("global error mode") → kept it local because boundaries are render-layer-only (correct exception).

**Generalization:** state lives in the reducer unless it's strictly render-layer concern (focus, scroll position, transient animation). When in doubt, push it to the reducer.

### 3. `instanceof` Discrimination Is Fragile But Practical

Story 2.4's network-vs-HTTP error discrimination uses `err instanceof TypeError`. Modern browsers (Chrome / FF / Safari / Edge / iOS Safari) all throw `TypeError` for fetch network failures; the apiClient throws plain `Error` for HTTP non-2xx. This works **today** but:

- Older browsers / polyfills / non-fetch environments may use plain `Error` for network failures.
- A programmer error elsewhere in the `.then` chain that throws TypeError gets misclassified.

The deferred-work backlog has 2 items related to broadening this detection (Story 2.4 review + Story 2.5 review). **Recommendation:** if a future story adds a true server-side bug or library upgrade that breaks `TypeError` discrimination, escalate to a custom `ApiHttpError` class or response-body inspection.

### 4. The Race-Condition Pattern Is Always: Capture-Then-Dispatch-Then-Schedule

Every Epic-2 race fix followed the same shape:

1. **Capture** the relevant reference / flag synchronously, BEFORE the dispatch.
2. **Dispatch** the optimistic action.
3. **Schedule** the side effect via rAF / `setImmediate` / `useEffect` AFTER React commits.

Examples: `loadCleanupRef` (Story 2.2), `retryInFlightRef` (Story 2.3), `fatalExit` + `setImmediate` (Story 2.5), focus-after-delete `fallback` capture (Story 2.6 Phase B1).

**Generalization:** when adding behavior that interacts with React's commit lifecycle, use the capture-dispatch-schedule pattern. The reducer dispatches synchronously; effects run after commit; refs survive both.

### 5. Story 2.6's `VERIFICATION.md` Is a Shipping Artifact, Not a CI Gate

The doc deliberately sits in the repo (not in CI). It's:

- An auditable record (date, build SHA, browsers tested).
- A regression baseline for future epics (bundle size, contrast pairs, axe-core findings, Lighthouse scores).
- A handoff document (the 10 `🟡 TBD-by-reviewer` rows are explicit work items for whoever runs the verification pass).

This is more durable than a CI dashboard and harder to game. The doc lives alongside the code; it travels with releases.
